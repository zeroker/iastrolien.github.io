Automatic memory leak detection on iOS(iOS自动检测内存泄漏)

在手机中，内存是共享资源。Apps 不合理的管理会使内存耗尽，崩溃，以及导致手机性能大幅度降低。
Facebook iOS app 拥有很多特性，他们共享一个内存空间。如果任何一个特性占用太多的内存，他将影响整个app的性能。例如：某个特性出现内存泄漏。
当我们给一组对象分配内存，使用完后没有释放，就会造成内存泄漏。这意味着操作系统没有办法回收内存并重新使用它们，内存终将耗尽。

在Facebook，许多工程师在不同的codebase中工作，内存泄漏是不可避免的。当这种情况发生时，我们需要快速的发现并修复它们。
现在已经存在许多检测内存泄漏的工具，但是他们需要大量的人工干预。
1.打开Xcode，选择build for profiling。
2.载入Instruments。
3.使用app，尽可能尝试重现场景和行为。
4.查看leaks/memory峰值。
5.查找内存泄漏的根源。
6.解决问题。

这意味着每次都需要大量的手工操作。因此，我们可能无法尽早在开发周期内定位已经修复内存泄漏。
如果该过程可以自动进行，我们就可以在更快的在没有太多开发者干预的情况下找到内存泄漏。
为了解决这个问题，我们已经构建了一系列工具来自动化这个过程和在我们的代码库中解决一些问题。今天，我们激动地宣布这些工具包括：FBRetainCycelDetector ,FBAllocationTracker,FBMemoryProfiler.

循环引用

Objective-C 使用引用计数管理内存，并释放不使用的对象。任何一个对象可以持有其他对象，只要需要就会将对象保持在内存当中。换句话说就是一个对象拥有另一个对象。
大多数情况下都会很好的工作。但是当两个对象互相持有的时候，直接或者间接的连接这两个对象，就会产生一种僵局。 这种持有方式叫做循环引用。

循环引用会引起一系列问题。最好的结果是内存在RAM中只占据一点点内存空间。如果泄漏的对象只是做一些不平凡的事情，那么其他app就只能使用更少的内存。最差的情况就是app泄漏的内存超过最大可使用内存，app就会崩溃。
在手动分析的过程中，我们会发现大量的循环引用。他们在早起开发的时候很容易出现循环引用，以后很难找到他们。Retain Cycle Detector 可以帮助我们很容易的找到他们。

运行时检测循环引用

在Objective-C中发现循环引用类似于在一个有向无环图中寻找环，结点就是对象，边就是对象间的引用(如果对象A持有对象B，那么A到B之间就存在引用)。Objective-C对象已经存在于我们的图中；我们使用深度优先遍历搜索他。

虽然这只是个简单的抽象，但是实际效果很好。我们得确保我们可以像使用节点一样使用对象，对于每一个对象，我们可以获得他所引用的所有对象。这些引用包括强引用(strong)和弱引用(weak)，循环引用是由于强引用(strong)导致的。对于每一个对象，我们需要知道如何找出这些强引用。
幸运的是，Objective-C提供了一套强大的运行时库，他可以给我们提供足够的数据去挖掘图。
在图中的一个节点可以一个对象或者一个闭包(block)。我们分开讨论

对象

在运行时有许多工具允许我们对对象进行内省学习。
我们首先要做的就是找到对象的实例变量布局(ivar layout) (详见runtime源码)

对于一个给定的对象，实例变量布局描述了我们应该去哪里查找被引用的其他对象。他将提供给我们一个索引，他代表偏移量，我们在对象的地址上加上偏移量来获得他所引用对象的地址。运行时还允许我们获得弱引用的实例变量的布局，我们假设这两种引用的布局的差别在于强引用布局。
他也支持部分Objective-C++，在Objective-C++中，我们可以定义一个对象结构体，但是在实例变量布局中不会找到这个结构体。运行时使用类型编码(type encoding)来解决这个问题。对于每一个实例变量，类型编码描述了变量是如何结构化的。如果变量是一个结构体，他描述了他包含的字段和类型。我们可以通过解析类型编码来获得那些事实例变量，那些事Objective-C对象。我们计算出他在布局中的偏移量，随后在不居中找到他指向对象的地址。
有一些边界情况我们不会深入探讨，大多数是一些不同的集合，我们可以遍历他们来获得他们持有的对象，这可能会有一些副作用。

Blocks

闭包是和对象有一点区别，在运行时中我们不能很容易看到他的布局，但是我们可以猜测他。
在处理blocks的时候，我们使用Mike Ash在Circle 项目中的思路，这个项目是在第一时间就使用FBRetainCycelDetector的。

我们可以使用ABI(application binary interface for blocks)，他描述了Block在内存中的布局。如果我们知道当前的引用是一个block，那么我们就可以使用一个结构体来模拟这个block对象，使用结构体后，我们就可以知道block对象持有的对象了。但是不行的是，我们不知道引用是强引用还是弱引用。
我们使用黑盒技术来解决这个问题。我们创建一个对象来伪装我们要研究的block对象。因为我们知道block的接口，也知道在哪里可以找到block持有的引用。伪造的对象使用”release detectors”来代替这些引用。release detectors是一些简单的对象，可以观察到发给他的release消息。当持有者想要放弃持有的对象的时候，消息就会发送给对象强引用的对象。在我们想释放这个release detectors的时候，我们可以检查detectors收到的release消息。知道了接收release消息的detectors的索引（在伪造对象中），就可以找到block对象所持有的强引用对象。

自动化

这些工具可以在工程师内部构建的时候可以连续并自动化运行。
客户端自动化很简单。我们在定时器安装循环引用检测器，定期扫描内存寻找循环引用。但是这并没有很顺利。我们第一次运行detector的时候，我们发现他无法扫描整个内存空间。刚开始我们需要提供一组候选对象供Detector扫描。
为了更高效处理上述问题，我们构建了FBAllocationTracker，这个工具可以记录NSObject对象及其子类的所有创建和销毁。他可以在任何时刻用很小的性能代价找到任何类的实例对象。
客户端有了上述自动化过程，我们只需要在NSTimer中使用FBRetainCycelDetector，并配合FBAllocationTracker来获取我们想要抓取的实例对象。

现在我们仔细看一下底层发生了什么。

循环引用可以包含任意数量的对象。由于一个坏的连接，导致了许多环，事情就会变得复杂很多。

A->B在环中就是一个坏的连接，他创建了两个环：A->B->C->D 和 A->B->C->E.
这会导致两个问题：
1.如果同一个坏连接导致两个环，我们不想用特殊的标记来标记他们。
2.我们不希望给两个环（代表不同的问题）同样的标记，尽管他们共享一个坏链。

所以我们需要给循环引用定义簇(clusters)，我们写了算法来找出泄漏问题：
1.在给定时间内收集所有的环。
2.对每个环，挑出facebook特定的类命名。
3.对每个环，找出环中包含的最小环。
4.将每个环添加到上述找到的最小化环所组成的组中。
5.只报告最小环。

最后一步就是找到谁能第一个偶然引入循环引用。我们可以对环涉及的代码使用’git/hg blame’，我们猜测他可能是最新的代码导致了这个问题。基于这个原因，我们可以给最后接触代码的人分配一个task，以便其修复问题。

手动分析

虽然自动化可以帮助我们简化发现循环引用的过程，减少开发人员的消耗，但是手动分析也还是必不可少的。我们构建另一个工具，允许任何人查看内存使用情况，你可以不用将手机插到电脑上就能查看。

FBMemoryProfiler可以很容易的添加到任何app，你可以在app中手动配置构建文件和运行循环引用检测。他是借助FBAllocationTracker和FBRetainCycelDetector来实现的。

Generations

FBMemoryProfiler的最大特性是”generation tracking”，类似于Instruments。Generations是两个时间标记中间所有仍然存活对象的简单快照。
使用带UI的FBMemoryProfiler，我们可以标记一个generation，比如：我们创建三个对象，然后我们标记另一个generation并继续创建对象。第一个generation包括我们创建的三个对象，如果任何一个对象被释放，那么他将在下第二个generation中被移除。

当我们有一个重复任务，我们认为他可能存在内存泄漏，此时generation tracking就可以派上用场了。例如：导航进一个试图控制器，然后再推出，在每次开始任务的时候我们标记一个generation，在每次generation标记之间进行查看。如果一个对象不应该一直活动，此时我们就可以在FBMemoryProfiler的界面中看到。

Check them out

无论你的app是大是小，功能是多是少，好的内存管理总是必要的。通过这些工具，我们能够轻松地找到并解决内存泄漏，所以我们可以花更少的时间在手动查找的过程，花更多的时间编写更好的代码。
